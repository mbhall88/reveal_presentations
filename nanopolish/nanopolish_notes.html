v0.2.0

Conceived of at a hackathon with Nick Loman and Josh Quick. Based on ideas from DALIGNER, Celera Assembler and poa. The methods for it’s first incarnation are outlined in the Nature Methods paper A complete bacterial genome assembled de novo using only nanopore sequencing data.

Nanopolish is an algorithm that uses the electrical current signal from nanopore sequencing to improve the consensus sequence of an assembly. The algorithm iteratively evaluates modifications to the sequence and calculates the probability of these modifications using a HMM.

They start with an initial draft assembly, , and progressively modify it by making small local changes. They then assess whether this modified version of , , increases the probability of seeing the sequence given the raw signal. If the probability increases, compared to , then is accepted.

Initially written in python, Jared quickly decided to rewrite in C++. The rewrite also removed the poa dependency. He had initially been using poa to compute the multiple sequence alignment and see the HMM realignment. However, in C++ he could use the initial multiple alignment directly from the BAM using htslib. With these two improvements [the other being C++ fast5 library] it took startup for the program from a few minutes to a few seconds.

Computing the Consensus Sequence

Uncorrected reads are mapped to a draft assembly using bwa mem. The genome is then segmented into 10Kbp segments that overlap by 200bp. An anchor is then selected every 50bp within a segment and is used to define a mapping between bases and the corresponding raw signal.

Prior to computing consensus, the transition probabilities for each strand are trained by aligning events for each strand to the draft assembly using the Viterbi algorithm (guided by the anchors). Event Align

Jared added a module into nanopolish to align between a reference genome and the raw signal of reads from that reference sample. The module is called eventalign and exposes a command-line interface for doing this alignment. It works by first aligning the reads to the reference (using bwa mem for the example, but it only needs a BAM file, so you could align with whatever you want). The output gives you a mapping from the reference position to the event mean for the read position that aligns to it. Note: If trying to label data for base caller training it might be more useful to have something that aligns this the next level through onto the actual raw signal. If the reference 5-mer was skipped there will be a gap in the output where no signal was recorded.

 

Nanopolish v0.4.0

The new sequencing kit, SQK006, from ONT changed from using 5-mers to 6-mers. As a result, data structures in nanopolish had to be updated to account for either K-mer.

Initial transition probabilities were also updated due to the increase in the speed of the pore from 30bp/sec to 70bp/sec. This increase caused more events to be missed.

dnadiff from MUMMR is used to calculate percent identity, the number of SNPs, and number of indels with respect to the reference (just thought I would mention this is it's good to know how other people are doing this calculation).

Loman did sequencing of their same K12 sample from the assembly paper with the new kit - one with native DNA and the other with PCR to remove damage and DNA modifications. The new kit gave better percent identity than the previous one and the PCR sample was slightly better than native (99.82% and 99.78%) despite PCR having lower coverage (30x) than native (40x).

eventalign also received a slight update. You can now get a summary file which will give the path to the fast5 file, the model name, number of aligned events, and other metadata for each read. There is also an optional --sam flag which will output an experimental SAM file containing the alignment of the event data to the reference. This has the advantage of being smaller in size than the standard .tsv file that eventalign produces and can also be used with htslib.

 

SNP calling

For the Ebola outbreak work Jared worked on SNP calling with nanopore data. The first (naive) attempt at this was effectively : If SNPs are well-separated we calculate the likelihood of a haplotype containing a SNP and compare this to the likelihood of the reference. One problem with this though is that you miscall regions with multiple SNPs nearby. Due to the sequencing being influenced by multiple bases nearby, you need to present a more "complete" haplotype to the likelihood function. Therefore you need to test SNPs in groups.

The first attempt was to a take a subsequence that we know (where the reads and reference agree) and then try to extend one base at a time by calculating . In other words: which base is most likely to extend the subsequence ? This approach turned out badly. If the extension made an error, that error tended to propagate and you end up with a tonne of false positives densely packed together.

The algorithm they ended up going with was as follows: Find candidate SNPs (mismatches between reads and reference with >20x coverage and 20% support) from nanopore reads aligned to the reference. The SNPs are batched into clusters that are separated by 10bp (Jared calculated this value and determined it to allow clusters to be treated independently). Generate all haplotypes ( for SNPs) and calculate the likelihood for each haplotype using the HMM. Highest likelihood is output to VCF if it is not the reference sequence. The algorithm is presented in the Ebola paper with Nick Loman - Real-time, portable genome sequencing for Ebola surveillance.

Candidate SNP generation

Iterate over the entire reference and examine positions with >20x coverage. Any mismatch that is seen at least 20% of the time is considered a candidate SNP.

Candidate haplotype generation

Due to the fact that the pore is influenced by more than a single base, they could not assess SNPs individually. Instead, the SNPs were clustered into subsets separated from other clusters by at least 10bp - these could be treated independently. For each subset all possible haplotype combinations are generated by including/excluding different combinations. Due to the number of combinations being (number of SNPs) subsets with more than 10 candidate SNPs, or spanning a region over 100bp were excluded. For each haplotype the likelihood of is calculated using the HMM from the polish method.

Haplotype likelihoods

The HMM calculates the probability of observing a sequence of events, , given a sequence . The model now also allows "soft-clipped" events to better handle uncertainty about where the event-to-sequence alignment starts and ends.

Variant calls

For each subset of candidate SNPs, the haplotype with the highest likelihood is called and all SNPs in the haplotype are output into a VCF. The score for the variant is the log likelihood ratio between the called haplotype and the reference haplotype.

Supporting R9 data

The release of R9 was a new pore and a new basecaller using an RNN instead of a HMM. Nanopolish was clearly going to need to change as it would need a new pore model (the model used to calculate emission distributions for the HMM), new transition probabilities (due to speed increase), and no longer relying on extracting scaling parameters as the RNN doesn't calculate these.

ONT provided a new table mapping kmers to Gaussian parameters. Jared wrote code to infer the per-read scaling parameters from the output of the RNN. The initial nanopolish results were not as good as R7 though. This is due to the increased speed, in addition to more aggressive event segmentation leading to more events than base movements (compared to the inverse with R7). More aggressive segmentation also leads to more noise getting into events, causing events to be very different from their model. To fix this, a new state was added to the HMM which would allow it to ignore events. Transitions to this state were given very low probability to avoid a "lazy" HMM wanting to ignore events. This new model increased consensus accuracy.

An interesting finding was that mixing R7 and R9 data lead to an even more improved consensus accuracy. This could be a direction to explore - mixed signal properties.

Homopolymers are still the biggest problem for consensus accuracy. However nanopolish is getting good results on 7-9bp homopolymers.

 

Nanopolish v0.6

This version of nanopolish brings methylation calling, no longer requires eventalign to do consensus and variant calling, models are now compiled with nanopolish (including methylation models for R7.3, R9, and R9.4), initial support for Albacore fast5 files, and experimental module for phasing reads given a VCF file.

The bottleneck in methylation calling at the moment is loading and parsing signal level information (not actually running the model).

This work was mostly outlined in the Nature Methods paper Detecting DNA cytosine methylation using nanopore sequencing.

Notes from paper

The method developed for detecting 5-mC methylation extends the HMM used by nanopolish already. To train the HMM they used PCR-amplified E. coli DNA that was either treated or untreated with the CpG methyltransferase M.SssI. They found the signal shift was dependent on the position of the 5-mC in a given kmer.

One limitation to the method at the moment is that it can't distinguish mixed methylation in close proximity. Therefore it assigns the same methylation status to CpG sites within 10bp of each other.

The initial method used a simple binary classifier to ask whether a site is methylated or not (sampled 100,000 singleton sites) and this achieved accuracy of 83% (R7.3) and 87% (R9). By being more stringent with their threshold they moved to >95% accuracy. This involved requiring that the absolute value of the log-likelihood ratio be above a threshold that was varied between 0 and 10. However , the increase in accuracy led to less sites being called. Setting the threshold to 2.5 gave 94% accuracy and called 77% of sites. This method was then used in replacement of the binary classifier.

Natural DNA can have DNA damage and 5-hydroxymethylcytosine which this model does not account for and so could cause a reduction in accuracy.

They then went on to explore the biological context around these methylation sites. Using the log-likelihood threshold of 2.5 they called methylation. They then assessed the percentage of called sites that were methylated as a function of their distance from an annotated transcription start site. As expected, they observed a tendency for sites near TSSs to be unmethylated (in natural DNA), with methylation increasing as you move further away from the TSS. This was corroborated by Illumina bisulphite sequencing.

They used a very neat method of control in training the new model. They took the unmethylated (PCR-amplified) samples and trained a model of all 6-mers and checked whether these differed noticeably from the models provided by ONT. They found that it was consistent with the ONT model. They then repeated this process, but instead they treated the PCR-amplified samples with MSssI methyltransferase. This gave very noticeable differences in some k-mers, with some differing by >4pA. It was concluded that these must be because of methylation.

When the methylated base was in the or position of the 6-mer they saw a consistent upwards shift in the signal. When in the first position however, they saw no change.

Nanopolish v0.8.0

Jared acknowledges that basecalling is moving away from event segmentation and towards end-to-end basecalling. Additionally, he noted that the event table stored in the fast5 files was fairly volatile and any dependencies between the k-mers in the event table and those in the reads would break the parser. The event table also takes up a huge amount of disk space. Some of the issues also made nanopolish incompatible with tools like poretools.

Nanopolish v0.8.0 can now directly read the raw data and segments it using the segmentation algorithm from scrappie. It can then calculate an alignment between the reads and events using dynamic programming. This is slightly slower (100 CPU hours to polish E. coli vs 85 using event table), but the offset means less disk space used and more robustness going forward.

 

Methylation-aware polishing

Added in functionality to nanopolish for methylation-aware polishing. If the candidate consensus sequence contains a methylation motif, such as GATC for Dam methyltransferase it will consider the possibility of the motif being methylated. To do this they have trained new models for Dam and Dcm (but mentions these are not ideal). Jared stressed this is still experimental and can be turned on with --methylation-aware=dam,dcm option in the variants command. Also, as this update required a substantial code refactor, a performance bottleneck was uncovered and fixed. This dropped runtime to 71 CPU hours on the E. coli dataset.

 

Nanopolish v0.9.0

nanopolish index has received a speed-up. If you provide it with the sequencing_summary.txt file it will index much faster. This prevents it having to open every fast5 file. On a test set it reduced index from 8 hours to 30 minutes!

The second speed-up also supports multi-threading input. Some workflows like call-methylation spent more time reading in fast5 files than they did calling methylation! The fast5 parsing code now is thread-safe and allows to remove the locks and process fast5 files in parallel. Methylation calling on human chromosome 20, using 8 threads, is now ~6 times faster. The consensus caller also benefits from this multi-threading.
